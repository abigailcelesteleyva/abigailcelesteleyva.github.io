Design system for the single ticket side pane view. Defines structure, hierarchy, behavior, and conditional rendering.

Layout & Hierarchy
Split layout: main content (left, scrollable) and metadata sidebar (right, 160px fixed). The main content contains everything the admin reads and acts on. The sidebar is reference metadata — always visible, never scrolls, slightly faded.

1. Header
The ticket's identity. Title left-aligned (17px), status and priority badges top-right. Ticket reference and source below the title, faded.

Source display depends on the integration type. Ticketing integrations include a linked external ID. Conversation integrations and email are plain text. Native shows no source line.

2. Metadata Sidebar
Right side, 160px fixed. Does not scroll. Slightly faded (opacity 0.75) to recede behind main content.

Consistent fields (always present): Requester, Assignee, Due — divider — Created, Last modified.

Conditional fields (max 3, from external source): Configured per integration + type, not per ticket. The admin builds muscle memory. Examples: SN Incident → Impact, Urgency, Configuration Item. Jira Bug → Component, Severity.

External source styling: Source logo + "[Integration] fields" header in source brand color. Subtle background tint. No opacity reduction — the tint is the signal.
The sidebar is for triage, not completeness. If the admin needs the full source view, the linked external ID in the header takes them there.

3. Enrichment
The agent's synthesis of everything it knows — from digital twin lookups, user conversations, and activity updates. Unlabeled in the UI. Position (immediately after header) and styling (full-size text, bold key facts) signal its importance.

For conversation integrations (Slack, Teams, native), the enrichment absorbs everything the user communicated. No separate description needed.

For ticketing integrations, the enrichment synthesizes data from the digital twin alongside whatever the external system provided. The original description appears separately below.
How to write the narrative
The narrative has a clear structure: what the user told us first, then what we found.

Start with the user's problem. Lead with what the person reported or is experiencing, in their terms. This is the situation as they see it. If they provided business context (a deadline, who's affected), include it here — it came from them, not from us.

Transition with "we found." Use natural language cues like "we found," "investigation shows," or "our checks revealed" to signal the shift from user-reported to agent-discovered. This is where the digital twin data appears.

Enrich the user's context with specifics. If the user said "payroll run is soon," the enrichment adds "affects 214 employees at 4:00 PM PST and she's the only admin with access." The user set the scene; we filled in the details they didn't have.

Bold the key facts that would change the admin's triage decision — numbers, system names, timeframes, affected scope. These are the datapoints that came from the digital twin.
Sources bar
A "Sources" bar sits below the enrichment showing which integrations were queried, with logos where available. Max 3 visible on one line — if more were queried, show 3 + clickable "+X more" that expands to reveal all. Click again to collapse.

4. Suggested Resolution
The agent's recommended workflow. Warm cream background — the only colored container in the pane, drawing the eye to the action. Workflow name as a 2-3 word link. Content and button vertically centered.

States: Idle (Run button), Running (bouncing STLabs logo), Completed (green logo, block fades), Failed (red logo + Retry or Investigate).

5. Resolution (Closed Tickets Only)
Agent-written summary of what actually resolved the ticket. Appears between enrichment and suggested resolution only when the ticket is closed. Labeled "RESOLUTION."

Adapts to however it was resolved: workflow ran (via pane or conversation), manual resolution with agent guidance, or resolved externally. Always a synthesis, never a transcript. No attribution ("Resolved by X") — the activity timeline has that detail.
Resolution
Workday payroll access restored via Workday Direct Unlock after initial MFA re-enrollment failed.
6. Description (Conditional)
Only when the ticket came from an external system — ticketing integrations (ServiceNow, Jira, etc.) or email. Uses the source system's field name: "DESCRIPTION · via [Integration]."

Not shown when the user communicated with our AI agent (Slack, Teams, native). The enrichment absorbs everything from that conversation.
If our agent was part of the conversation that created the ticket, description is absorbed into enrichment. If it came from a form the agent wasn't part of, it's shown separately.
Description · via ServiceNow
I can't get into Workday. I've tried logging in three times and the MFA push isn't coming through. We have a scheduled pay run in 4 hours for the full company.
Smaller, muted — reference material, not primary content.
7. Activity Timeline
Collapsible timeline at the bottom. Collapsed by default with chevron + event count. Newest first. Three actor types: Source, Agent, JK Human. Plus outcomes.
See: Activity Timeline Design System →
Conditional Rendering
Native
Header (ID only) → Enrichment → Suggested resolution → Activity. No description. No sidebar tint.
Conversation
Slack, Teams, etc. Header (no link) → Enrichment → Suggested resolution → Activity. No description — enrichment absorbs the conversation. No sidebar tint.
Ticketing
ServiceNow, Jira, etc. Header (linked ID) → Enrichment → Suggested resolution → Description (· via [Integration]) → Activity. Sidebar tinted with "[Integration] fields" header.
Email
Header (via Email) → Enrichment → Suggested resolution → Description (email body) → Activity. No sidebar tint.
Closed
Adds Resolution between Enrichment and Description. Suggested resolution collapses away (smooth height + fade animation). Status badge → grey "Closed." The workflow record lives in the activity timeline.
