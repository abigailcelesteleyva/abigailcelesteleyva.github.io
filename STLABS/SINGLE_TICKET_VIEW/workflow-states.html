Workflow Run — States

How the suggested resolution block behaves across the full lifecycle of a workflow run.

Design Principles
Three states: idle, running, terminal
A workflow is either waiting to run, actively running, or done. "Done" splits into two outcomes: completed and failed. There is no paused or queued state in the single-ticket view — those exist at the group/batch level only.
The logo is the running indicator
While running, the STLabs logo gently pulses and bounces in place of the Run button. No spinner, no progress bar, no "Running" text. The brand doing the work is the signal. The animation is slow (1.2s) and subtle — calm, not urgent.
Completed = green logo + "Completed"
On success, the logo turns green and "Completed" text appears. No checkmark icon — the green logo is enough. The block fades to muted background to signal "done, no longer actionable."
Failed = "Failed" text + Retry, Investigate, or agent auto-suggests
On failure, small "Failed" text appears upper-right with STLabs logo (red-tinted). For transient errors, a Retry button appears below it. What happens after retry fails (or on deterministic errors) depends on context — see next principle. No error message in the block — the timeline has the detail.
Context decides the next action
The admin doesn't guess. First failure on a transient error shows Retry. After that:

In a conversation with the agent — the agent auto-suggests a new workflow in chat. The suggested resolution block swaps with an animation. No button needed; the agent drives it.

Not in a conversation — an Investigate button (with chat icon) appears, opening the agent chat. From there the agent can suggest alternatives, guide manual resolution, or escalate.
Idle and Running are vertically centered. Failed is top-aligned.
The Run button and running logo are vertically centered against the text — feels balanced. Failed states are top-right aligned — the "Failed" label and action button sit at the top of the block, compact and out of the way of the workflow description.
Timeline reflects the outcome, not the process
The activity timeline gets one event for the run ("Jordan K. ran Okta MFA Reset") and one for the outcome ("Okta MFA Reset completed" or "Okta MFA Reset failed — reason"). No intermediate steps. Failures use a red dot in the timeline.

Lifecycle
Happy path
Idle → Running (1–3s) → Completed. Block fades. Resolution section appears above. Status badge changes to "Closed." Timeline gets: "[User] ran [workflow]" + "[Workflow] completed" + "Ticket closed."
Failure → Retry (when it makes sense)
Idle → Running → Failed (retryable). Block shows "Failed" upper-right with Retry button. Only for transient errors: timeout, 503, rate limit — things that might work on a second attempt. Timeline gets: "[User] ran [workflow]" + "[Workflow] failed — [reason]" (red dot). Max 1 retry before escalation.
In conversation: agent suggests new workflow
When the admin is already chatting with the agent and a retry fails, the agent automatically suggests a new workflow. The suggested resolution block animates (workflow-swap: slide up, brief orange glow, settle) to show the new workflow with a fresh Run button. No Investigate button — the conversation is already open. The failed workflow stays in the timeline as a record.
Not in conversation: Investigate opens chat
When the admin ran the workflow from the side pane without an active chat, failed retry or deterministic errors show an Investigate button with a chat icon. Clicking it opens the agent conversation. From there the agent can: (1) Suggest a different workflow — the resolution block swaps. (2) Guide manual resolution via chat. (3) Escalate further — assign to a specialist or create a linked ticket.
