Design Decisions

Key decisions and their rationale. Each decision shaped how the ticket pane works today.

v1.0 · March 2026

Information Architecture
Decision 1
Enrichment replaces description for conversation sources
When a ticket comes from Slack, Teams, or native chat where the user talked to our agent, the enrichment absorbs everything. No separate description needed — the agent was part of the conversation and synthesizes it all. Description only appears for external ticketing systems (ServiceNow, Jira) where we received a form submission the agent wasn't part of.
Decision 2
No label on the enrichment
The enrichment is the hero content — it doesn't need a label. Position (immediately after header) and styling (full-size text, bold key facts) signal its importance. We tried "Context", "What we know", "Summary" — all felt reductive. The content speaks for itself.
Decision 3
The "we found" transition
The enrichment narrative has a clear structure: what the user told us, then what we found. Natural language cues like "we found" or "investigation shows" signal the shift from user-reported to agent-discovered. This lets the admin distinguish between what came from the person vs what the agent looked up.
Decision 4
Sources bar instead of "Checked"
Originally labeled "Checked" — changed to "Sources" to be descriptive rather than action-oriented. Shows which integrations were queried with logos. Max 3 visible + expandable "+X more" to avoid visual noise.
Decision 5
Resolution disappears the suggested resolution
When a workflow completes and the resolution section appears, the suggested resolution block smoothly collapses away (cross-fade animation). Showing both is redundant — the resolution tells you what happened, the activity timeline has the full record. The completed state shows briefly (green logo + "Completed") before the transition.
Decision 6
Resolution has no attribution
Resolution text doesn't include "Resolved by X" — the activity timeline has that detail. The resolution is about what happened, not who did it.
Metadata & External Sources
Decision 7
Source zone over per-field labels
For external ticketing integrations (ServiceNow, Jira), fields from the source system are grouped in a tinted zone with the integration logo. We explored per-field blue labels first — they were noisy and scattered attention. The zone approach is calmer: one visual region, one logo, immediately scannable. The admin builds muscle memory for where external fields live.
Decision 8
Sidebar metadata (right side, 160px) over horizontal strip
Metadata moved from a horizontal strip below the header to a fixed sidebar on the right. The strip competed with content for vertical space. The sidebar is always visible, never scrolls, and recedes behind the main content at 75% opacity. Conditional fields (max 3 from external source) are configured per integration+type, not per ticket.
Decision 9
Edited badges stay, strikethrough values fade
When the agent or user edits a field, the [STLabs logo] Edited badge persists but the strikethrough previous value fades away after the edit conversation moves on. Keeps the signal ("this was changed") without the noise of old values cluttering the sidebar.
Activity Timeline
Decision 10
Agent assessed ticket — decisions always visible
Renamed from "Agent processed ticket" to "Agent assessed ticket" — implies evaluation + judgment. The key decisions (suggested workflow + assignment) show as a visible summary line below. "Show more" only reveals the research trail (queried sources). Most users never need to expand.
Decision 11
Three actors + outcomes
Timeline has three actor types: Source (integration logo), Agent (greyed STLabs logo), Human (initials avatar). Plus outcomes (neutral/red dots). The spine icon always shows WHO. The text shows WHAT and WHERE. Simple, scannable, consistent.
Decision 12
External comments use the person as the actor, not the system
When Sarah K. comments via ServiceNow, the timeline shows her avatar (SK) with a left-bordered bubble containing the ServiceNow logo. The actor is always a person — the system is just the channel.
Decision 13
Agent commenting on external source
When the agent posts a comment to ServiceNow, it uses the standard agent communication pattern: greyed STLabs icon + "Agent commented via ServiceNow" + message bubble. No special email/external chrome — stays consistent with the design system. We'll rev on email treatment separately.
Workflow Run States
Decision 14
Context decides what happens after failure
In a conversation: agent auto-suggests a new workflow (block swaps with animation). Not in a conversation: Investigate button opens chat. The admin never guesses — the system decides Retry vs new workflow vs Investigate based on error type and context.
Decision 15
The logo is the running indicator
While running, the STLabs logo gently pulses and bounces (1.2s cycle). No spinner, no progress bar, no "Running" text. The brand doing the work is the signal.
Decision 16
Suggested resolution collapses with a cross-fade
When workflow completes: running logo → completed state (green logo, 700ms) → suggested resolution collapses out while resolution section fades in. One continuous motion using Framer Motion's AnimatePresence mode="popLayout". No discrete steps.
